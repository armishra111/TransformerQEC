# TransformerQEC

## Project Overview
TransformerQEC is a machine learning research project that trains a parameter-efficient (1.3M parameter) Physics-Informed Transformer Decoder for Quantum Error Correction on Rotated Surface Codes. It evaluates a JAX/Flax-based Transformer model against the Minimum Weight Perfect Matching (MWPM) baseline using synthetic syndrome data generated by STIM under a phenomenological noise model. The project focuses on parameter efficiency, custom topological inductive biases like (2+1)D Anisotropic RoPE, and hardware alignment (TPU targeting).

## Building and Running
*   **Environment (TRC Cloud TPU):** This project is designed to run on a Google Cloud Project utilizing Cloud TPUs (e.g., v3-8, v4-8) provided by the TPU Research Cloud (TRC) program. You will need a GCP project with the Cloud TPU API enabled and a TRC-granted TPU quota. Connect to your Cloud TPU VM via SSH.
*   **Dependencies:** The project dependencies are pinned in `requirements.txt`. On a Cloud TPU VM, ensure JAX is installed with TPU support before installing the remaining dependencies:
    ```bash
    # Install JAX with TPU support (often pre-installed on TPU VMs, but good to verify)
    pip install -U "jax[tpu]" -f https://storage.googleapis.com/jax-releases/libtpu_releases.html
    pip install -r requirements.txt
    ```
*   **Training and Evaluation:** The main workflows are orchestrated via Jupyter notebooks in the `notebooks/` directory. You can run these on your Cloud TPU VM by starting a Jupyter server (`jupyter lab --ip=0.0.0.0 --no-browser`) and connecting via SSH port forwarding, or by exporting them to python scripts.
    *   `notebooks/01_data_exploration.ipynb`: STIM circuit inspection and noise model characterization.
    *   `notebooks/02_model_and_training.ipynb`: Model architecture (imported from `notebooks/model.py`), on-the-fly STIM sampling, and TPU training loop.
    *   `notebooks/03_evaluation.ipynb`: MWPM baseline comparison, Wilson 95% CI bands, threshold estimation, and plotting.
*   **Testing:** End-to-end integration and equivariance regression tests are run using pytest.
    ```bash
    pytest tests/
    ```

## Development Conventions
*   **Role Constraint (CRITICAL):** According to `PLAN.md` and `STATE.md`, **Gemini models MUST NOT modify code.** Gemini is strictly limited to planning, research documentation, and collaborative analysis. Only Claude-code is authorized for code refactoring, structural implementation, and direct file modifications.
*   **Architecture & Efficiency:** The core constraint is parameter efficiency (maintain success at ~1.3M parameters). Future architectures must prioritize elegant, physics-informed inductive biases (like symmetries and (2+1)D positional embeddings) over brute-force data scaling.
*   **Tracking Progress:** The project roadmap, current status, constraints, and actionable insights are rigorously tracked in `PLAN.md` and `STATE.md`. Always review these files to understand the current research cycle before brainstorming, and ensure they are updated to reflect the latest status at the end of a session.
*   **Frameworks:** JAX/Flax for the model architecture, STIM for high-speed quantum circuit simulation and dataset generation, and PyMatching for the MWPM classical baseline.
*   **Precision & Optimization:** Enforce precision discipline (e.g., `bfloat16` for Q-K matmuls, upcast to `float32` for `softmax`) and hardware-aware optimizations like gradient checkpointing (`nn.remat`) to maximize TPU throughput.
